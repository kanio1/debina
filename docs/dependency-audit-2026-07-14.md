# Dependency Audit — 2026-07-14

Iteration 0 maintenance gate performed before continuing epic work (EPIC-12+). Scope: backend Maven ecosystem, frontend npm/pnpm ecosystem, security advisories for both. All version claims below are backed by a primary source (Maven Central / central.sonatype.com, npm registry, official GitHub releases, or `gh api`) — no MvnRepository, blog, or AI-summarized version claims were used as the source of truth for a decision.

## Executive summary

- **Backend**: found and fixed a real version-drift bug — `spring-modulith.version` was pinned to `2.0.6` (the line compiled/tested against Spring Boot **4.0**), while the project already runs Spring Boot **4.1.0**. Spring Modulith `2.1.0` GA (2026-06-11) is the version whose own release notes state "Upgrade to Spring Boot 4.1" — bumped to `2.1.0`. Also found a manual `junit-jupiter` version override (`5.11.0`) fighting Spring Boot's own BOM, which manages JUnit at `6.0.3` — this produced two JUnit versions resolving on the classpath simultaneously (a real convergence defect, confirmed via `dependency:analyze`/`dependency:tree`). Removed the override; JUnit now resolves cleanly to the BOM-managed `6.0.3` (JUnit 6, GA, no functional break — full regression re-run confirms all 26 tests still pass). Bumped `postgresql` JDBC driver `42.7.7` → `42.7.13` (patch only). Flyway, Testcontainers, and ArchUnit (transitive) were already at latest stable and needed no change.
- **Frontend**: found and fixed a real, confirmed-exploitable **moderate security vulnerability** (`pnpm audit`): `postcss@8.4.31`, bundled transitively by `next@16.2.10` itself, is vulnerable to `GHSA-qx2v-qp2m-jg93` (XSS via unescaped `</style>` in CSS stringify output) for versions `<8.5.10`. Next.js has no newer patch to pull in a fixed copy (16.2.10 is already latest stable), so added a `pnpm.overrides` pin forcing `postcss` to `8.5.19` (already present elsewhere in the tree via `tailwindcss`/`shadcn`) everywhere — eliminates the vulnerable copy without touching Next.js itself. `pnpm audit` is now clean (0 vulnerabilities). Bumped `react`/`react-dom` `19.2.4` → `19.2.7` (patch). Bumped `@types/node` `^20` → `^24` — the previous `^20` range was a leftover scaffold default that didn't track the project's own exact Node pin (`24.18.0`, per `CLAUDE.md`/`frontend/.node-version`); typing against Node 20's API surface while running Node 24.18.0 was itself a latent correctness gap, not a stylistic one.
- **Security**: 1 finding (frontend, moderate, `postcss`), fixed. No high/critical findings in either ecosystem.
- **Deferred**: `eslint` `9.39.5` → `10.7.0` (major, no forcing security issue).
- **Update (2026-07-14, later session)**: `typescript` was re-audited and **pinned to `6.0.3`**. The prior claim in this report's first pass — "TypeScript skipped a numbered 6.0 GA entirely" — was **wrong**, and is corrected here rather than left standing. Direct re-check against the npm registry (`npm view typescript versions --json`, `npm view typescript dist-tags --json`) shows `6.0.2` and `6.0.3` are real, non-prerelease, published GA versions, distinct from the `6.0.0-beta`/`6.0.0-dev.*`/`6.0.1-rc` prereleases that came before them. The registry's `latest` dist-tag has simply already moved on to `7.0.2`, which is why a check that only reads the `latest` tag (as the first pass did) misses the 6.x GA line entirely — an incomplete-source mistake, not a real absence. Compatibility confirmed before pinning: `typescript-eslint@8.64.0` (latest stable, and what this project actually resolves) declares `peerDependencies.typescript: ">=4.8.4 <6.1.0"`, which includes `6.0.3`; `eslint-config-next@16.2.10` only requires `typescript >=3.3.1`; `next@16.2.10` has no `typescript` peer dependency. No TypeScript 6.0 breaking change (removal of `moduleResolution: classic`; forced ESM interop) affects this project's `tsconfig.json` (already `"moduleResolution": "bundler"`, `"esModuleInterop": true`). Full regression clean: `pnpm exec tsc --version` → `6.0.3`; `lint`/`typecheck`/`build` clean; `pnpm audit` → 0 vulnerabilities; `pnpm install --frozen-lockfile` → PASS. See `frontend/README.md`'s "TypeScript version pin" section for the same evidence, kept in sync.
- All groups verified with a real regression after each change: backend tests, frontend `lint`/`typecheck`/`build` clean, `act` gates for both workflows PASS, `git diff --check` clean.
