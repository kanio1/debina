# Dagger Pipeline Architecture

## Status

Phase D is at `TOOLING-BLOCKED-CHECKPOINT`: neither the Dagger CLI nor Go is installed on the verified workstation. No Dagger module has been created because the module format, generated bindings and `check` surface must be generated by the installed stable CLI, not guessed. See [the toolchain baseline](DAGGER-TOOLCHAIN-BASELINE.md).

## Intended local architecture after the gate passes

One provider-neutral Go SDK module will own local composition only. The repository commands and tests remain authoritative leaves. Public no-argument Dagger checks will be `fast`, `integration`, and `smoke`; unfiltered `dagger check` is the canonical complete set. An `all` alias is conditional on the installed CLI proving it does not duplicate or recurse.

| Group | Intended membership |
|---|---|
| `fast` | governance runner; Dagger self-tests; selected backend compile/architecture/focused tests; frontend codegen direct-drift comparison, lint and typecheck |
| `integration` | full Maven regression, Testcontainers compatibility, frontend build, fresh/upgrade Flyway, pipeline RLS/grant probes and Kafka readiness |
| `smoke` | ephemeral PostgreSQL, Kafka, Keycloak, backend, Next.js BFF and the six ADR-N16 Chromium journeys |

The module must compose Go functions and the Dagger graph directly: it must never invoke `dagger check`, `dagger call` or `dagger run` from inside a function. Source inputs will use the installed SDK's workspace mechanism with tight exclusions, preserving authoritative generated GraphQL and Dagger bindings while excluding `.git`, `build`, dependency/build outputs and test reports.

Remote CI, `act`, deployment, publishing and release automation remain deferred. No provider is selected.
